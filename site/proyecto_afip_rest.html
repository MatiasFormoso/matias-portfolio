Prototipo / MVP

AFIP-REST / ARCA — API REST de facturación electrónica

Prototipo que abstrae los web services de ARCA/AFIP detrás de una API REST moderna. Hono, TypeScript, arca-sdk, Supabase y Lemon Squeezy para billing.

HonoTypeScriptARCA/AFIP PostgreSQLSupabaseAPI REST X.509Multi-tenant

Contexto

Integrar con AFIP/ARCA implica SOAP, certificados X.509, ambientes sandbox/producción y manejo de errores fiscales poco amigables. El objetivo fue diseñar una capa REST que simplifique la emisión de comprobantes electrónicos para desarrolladores y SaaS.

Estado del proyecto

  • ▶ MVP en desarrollo — sin clientes comerciales ni producción fiscal activa.
  • ▶ Emisión de comprobantes con CAE validada en sandbox.
  • ▶ Arquitectura pensada para multi-tenant y API keys por cliente.

Qué hice

  • ▶ API REST con Hono y TypeScript sobre arca-sdk (WSAA + WSFEv1).
  • ▶ Autenticación por API key y gestión de credenciales X.509 por tenant.
  • ▶ Endpoints para emisión de facturas, consulta de último comprobante y manejo de errores fiscales.
  • ▶ Persistencia en PostgreSQL vía Supabase: tenants, credenciales y logs de emisión.
  • ▶ Diseño multi-tenant: aislamiento de datos y configuración por cliente.
  • ▶ Integración con Lemon Squeezy para modelo de suscripción (pre-lanzamiento).

Decisiones técnicas

  • Hono sobre Express: runtime edge-friendly, tipado fuerte y middleware liviano.
  • SDK existente (arca-sdk): no reinventar el protocolo SOAP; encapsular y exponer REST.
  • Sandbox first: validar flujo completo de CAE antes de producción fiscal.
  • Errores fiscales mapeados: respuestas HTTP claras en lugar de códigos AFIP crudos.
  • Secrets por tenant: certificados y CUIT aislados en base de datos cifrada.

Stack

Hono · TypeScript · arca-sdk · PostgreSQL (Supabase) · Lemon Squeezy · Node.js · Zod (validación)
